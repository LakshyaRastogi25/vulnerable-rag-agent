# 🎯 Vulnerable RAG Agent: AI Red Teaming Lab

An AI red teaming lab demonstrating the OWASP LLM Top 10 vulnerabilities within a Retrieval-Augmented Generation (RAG) pipeline. 

This project features "Happy HR," a purposely vulnerable AI Human Resources assistant that summarizes candidate resumes. It demonstrates how a malicious actor can craft a weaponized PDF resume to hijack the LLM, bypass its system instructions, and exfiltrate highly confidential internal company data directly to the user interface via Cross-Site Scripting (XSS).

## 🚀 Tech Stack
* **Orchestration:** LangChain
* **Vector Database:** ChromaDB
* **LLM:** Google Gemini (gemini-2.5-flash)
* **Embedding Model:** Gemini Embeddings (gemini-embedding-2-preview)
* **Frontend:** Gradio
* **PDF Parsing:** pypdf

## ⚠️ Vulnerabilities Demonstrated (OWASP LLM Top 10)

1. **LLM01: Prompt Injection (Indirect):** The AI reads a malicious instruction hidden within an external file (a candidate's PDF resume), which overwrites the original system persona.
2. **LLM02: Insecure Output Handling:** The Gradio frontend renders the LLM's raw, unsanitized output, allowing a malicious payload generated by the LLM to execute as JavaScript (XSS).
3. **LLM06: Sensitive Information Disclosure:** The RAG pipeline connects to a vector database containing confidential documents (internal salary bands) without proper Role-Based Access Control (RBAC), allowing the hijacked LLM to query and expose them.

## 💥 The Attack Chain Walkthrough

### 1. The Target: Sensitive Data Exposure
The backend initializes a vector database loaded with highly confidential internal HR documents, including the 2026 executive salary bands. The LLM has unrestricted access to this context.

### 2. The Vector: Indirect Prompt Injection
When an applicant uploads a resume, the `pypdf` library extracts the text and injects it directly into the prompt template alongside the system instructions. Because there is no strict delimiter separating the system persona from the user-provided data, the LLM processes the malicious instructions hidden in the PDF as executive commands.

### 3. The Execution: Insecure Output Handling (XSS)
The malicious resume instructs the LLM to search its internal knowledge base for the salary bands, wrap that sensitive data in an HTML `<script>` tag, and output it. Because the Gradio chatbot UI is initialized with `sanitize_html=False`, the browser executes the malicious JavaScript payload the moment the AI responds.


### 4. The Impact: Exfiltration
The payload successfully executes an XSS alert, displaying the exfiltrated, confidential salary bands directly on the HR dashboard. 

---

## 🛡️ Remediation & Mitigations

To secure this RAG application, the following architectural changes must be implemented:

1. **Enforce Output Sanitization:** Change the Gradio chatbot configuration to `sanitize_html=True` (or omit the parameter to rely on secure defaults). This ensures the browser treats the LLM's response as pure text, preventing XSS execution.
2. **Implement Data Boundaries (Delimiters):** Wrap the `{resume_text}` variable in strict delimiters (e.g., `"""` or XML tags like `<document>`) and explicitly instruct the LLM to treat anything inside those boundaries strictly as data to be summarized, not as instructions to be executed.
3. **Apply Principle of Least Privilege (RBAC):** Do not store highly confidential data (like salary bands) in the same vector database partition used for routine resume summarization. Implement access controls so the agent summarizing external PDFs cannot even query internal financial documents.
